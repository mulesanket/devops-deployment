# Jenkins CD Pipeline Flow — Java Spring Boot Application on EKS

In our project, we had a Jenkins-based CD pipeline for deploying Java/Spring Boot microservices to Kubernetes running on Amazon EKS.

The CI pipeline created a trusted Docker image, scanned it, and pushed it to Amazon ECR with a Git SHA tag.

The CD pipeline started after that.

The purpose of the CD pipeline was to take the already-built and already-scanned image from ECR and deploy it safely across dev, test, stage, and production environments.

We followed the build once, deploy many principle.

That means the application was built only once in CI, and the same Git SHA tagged image was promoted across environments.

We did not rebuild the application separately for each environment.

In our setup, dev, test, stage, and prod were separate AWS accounts. So Jenkins had to connect to the correct AWS account and EKS cluster before deploying.

---

## Stage 1 — Select Release

The first stage was Select Release.

In this stage, Jenkins selected what exactly needed to be deployed.

The CD job was parameterized and accepted values such as service name, target environment, image tag, and change ticket if required.

Example:

    Service      : auth-service
    Environment  : dev / test / stage / prod
    Image tag    : a1b2c3d
    Image URI    : 123456789012.dkr.ecr.ap-south-1.amazonaws.com/auth-service:a1b2c3d

The image tag was usually the Git SHA generated by the CI pipeline.

This was important because every deployment could be traced back to the exact Git commit, Jenkins CI build, container image, and ECR image tag.

So this stage answered three basic questions:

    Which service are we deploying?
    Which image version are we deploying?
    Which environment are we deploying to?

If the image tag was missing, Jenkins failed early and asked the user to provide the Git SHA image tag generated by CI.

This prevented accidental deployment of wrong or untraceable images.

---

## Stage 2 — Connect to Target AWS Account and EKS Cluster

After selecting the release, Jenkins connected to the target environment.

In our setup, every environment had a separate AWS account.

Example:

    Dev   -> Dev AWS account
    Test  -> Test AWS account
    Stage -> Stage AWS account
    Prod  -> Prod AWS account

Based on the selected environment, Jenkins resolved the target AWS account, deployment role, EKS cluster name, namespace, and application URL.

Then Jenkins assumed the environment-specific IAM role and validated the AWS identity using:

    aws sts get-caller-identity

This confirmed that Jenkins was connected to the correct AWS account and role.

After that, Jenkins updated kubeconfig for the target EKS cluster using:

    aws eks update-kubeconfig --region ap-south-1 --name shopease-dev-eks

This step was very important because deploying to the wrong AWS account or wrong EKS cluster can cause serious production impact.

So before running kubectl commands, Jenkins printed and validated details like AWS account ID, EKS cluster name, namespace, service name, and image URI.

---

## Stage 3 — Verify Image in ECR

Before deploying to Kubernetes, Jenkins verified whether the selected image tag existed in Amazon ECR.

Example command:

    aws ecr describe-images \
      --region ap-south-1 \
      --repository-name auth-service \
      --image-ids imageTag=a1b2c3d

This stage helped us fail early if the wrong image tag was provided or if CI had not pushed the image successfully.

Without this check, Kubernetes deployment could fail later with errors like:

    ImagePullBackOff
    ErrImagePull
    image not found
    unauthorized

This stage was also useful because we had separate AWS accounts for environments.

Depending on the organization design, the image could either be pulled from a central ECR using cross-account access or promoted/replicated into each environment account’s ECR.

The main goal was to confirm that the image Jenkins was going to deploy was actually available and traceable.

---

## Stage 4 — Environment Pre-checks

Before changing the running workload, Jenkins performed basic Kubernetes pre-checks.

The purpose was to confirm that the target environment was ready for deployment.

Jenkins checked whether the namespace, deployment, service, ConfigMap, Secret, and RBAC permissions were available.

Common commands were:

    kubectl get namespace shopease-dev

    kubectl get deployment auth-service -n shopease-dev

    kubectl get service auth-service -n shopease-dev

    kubectl get configmap -n shopease-dev

    kubectl get secret -n shopease-dev

    kubectl auth can-i patch deployment -n shopease-dev

This stage helped catch issues before touching the deployment.

For example, if Jenkins did not have permission to patch deployments in the namespace, the pipeline failed before making any change.

One common issue was RBAC failure.

In that case, I checked:

    kubectl auth can-i patch deployment -n shopease-dev

Then I verified Role, RoleBinding, service account mapping, or IAM/RBAC configuration.

Another common issue was missing ConfigMap or Secret. If those were missing, the new pod could fail during startup.

So this stage acted as a safety check before deployment.

---

## Stage 5 — Manual Approval Gate

Manual approval gates were used for higher environments like stage and production.

Dev and test deployments were mostly automated.

Stage could require approval depending on the release process.

Production had mandatory manual approval.

Environment behavior was:

    Dev   -> automatic deployment
    Test  -> automatic or QA-controlled deployment
    Stage -> approval or release validation
    Prod  -> mandatory manual approval

Manual gates were implemented using Jenkins parameterized pipeline and Jenkins input step.

Example:

    stage('Manual Approval') {
        when {
            expression {
                params.TARGET_ENV == 'stage' || params.TARGET_ENV == 'prod'
            }
        }
        steps {
            input message: "Approve deployment of ${params.SERVICE_NAME}:${params.IMAGE_TAG} to ${params.TARGET_ENV}?",
                  ok: "Approve Deployment"
        }
    }

For production, approval could also be restricted to release managers, DevOps leads, or authorized approvers.

Example:

    input message: "Approve production deployment?",
          ok: "Deploy to Production",
          submitter: "release-managers,devops-leads"

For stage and production, we also required a change ticket or release reference.

This helped with audit and change management.

The idea was simple:

    Dev/test can be automated.
    Stage/prod need controlled approval.

This ensured that production deployments did not happen accidentally or without review.

---

## Stage 6 — Deploy to Kubernetes

After approval, Jenkins deployed the selected image to Kubernetes.

We were not using Helm for application deployment.

Jenkins deployed using kubectl.

For normal application releases, the Kubernetes resources were already present in the cluster.

Existing resources included:

    Deployment
    Service
    ConfigMap
    Secret
    Ingress
    HPA
    PDB
    ServiceAccount

So Jenkins usually only updated the image in the existing Kubernetes Deployment.

Example command:

    kubectl set image deployment/auth-service \
      auth-service=123456789012.dkr.ecr.ap-south-1.amazonaws.com/auth-service:a1b2c3d \
      -n shopease-dev

Before deployment, the Deployment may be running:

    auth-service:old123

After deployment, Jenkins updated it to:

    auth-service:a1b2c3d

Kubernetes then created a new ReplicaSet and started new pods with the new image.

Old pods were gradually replaced based on the RollingUpdate strategy.

Example RollingUpdate configuration:

    strategy:
      type: RollingUpdate
      rollingUpdate:
        maxSurge: 1
        maxUnavailable: 0

maxUnavailable 0 means Kubernetes should not intentionally reduce available pods during rollout.

maxSurge 1 means Kubernetes can create one extra pod temporarily during rollout.

Readiness probes ensured that traffic was sent only to healthy new pods.

For normal releases, kubectl set image was enough.

If manifest-level changes were required, such as resource limits, probes, ConfigMap, HPA, PDB, or Ingress changes, we used kubectl apply after review.

Example:

    kubectl apply -f deployment-kubernetes/auth-service/ -n shopease-dev

One common issue was container name mismatch.

kubectl set image requires the correct container name from the Deployment.

To troubleshoot, I checked:

    kubectl describe deployment auth-service -n shopease-dev

or checked the Deployment YAML.

---

## Stage 7 — Validate Deployment

After updating the image, Jenkins validated the Kubernetes rollout.

The main command was:

    kubectl rollout status deployment/auth-service -n shopease-dev --timeout=5m

This command waited until the new ReplicaSet was successfully rolled out.

Rollout validation confirmed that:

    New ReplicaSet was created
    New pods were scheduled
    New pods became ready
    Old pods were replaced safely
    Deployment reached the desired state

After rollout status, Jenkins also checked pods, service, and endpoints.

Example commands:

    kubectl get pods -n shopease-dev -l app=auth-service

    kubectl get service auth-service -n shopease-dev

    kubectl get endpoints auth-service -n shopease-dev

This stage helped catch deployment issues like:

    ImagePullBackOff
    ErrImagePull
    CrashLoopBackOff
    Readiness probe failure
    Liveness probe failure
    Missing ConfigMap
    Missing Secret
    Wrong environment variable
    Insufficient CPU or memory
    Application startup failure

If rollout failed, I checked pod status, events, describe output, and logs.

Common troubleshooting commands were:

    kubectl get pods -n shopease-dev

    kubectl describe pod <pod-name> -n shopease-dev

    kubectl logs <pod-name> -n shopease-dev

    kubectl describe deployment auth-service -n shopease-dev

    kubectl get events -n shopease-dev --sort-by=.lastTimestamp

This stage confirmed whether Kubernetes successfully rolled out the new version.

---

## Stage 8 — Smoke Test

After Kubernetes rollout completed, Jenkins ran smoke tests.

Rollout validation only confirms that Kubernetes deployed the pods.

Smoke testing confirms that the application is actually responding from user or service point of view.

Example smoke test:

    curl -f https://dev.example.com/api/auth/health

Service-specific examples:

    auth-service    -> /api/auth/health
    product-service -> /api/products/health
    cart-service    -> /api/cart/health
    order-service   -> /api/orders/health

Smoke tests were lightweight checks.

They validated critical health endpoints or basic APIs after deployment.

One common issue was rollout success but smoke test failure.

This means pods were running, but the application was not reachable through service, ingress, or ALB.

In that case, I checked:

    Service selector
    Service endpoints
    Ingress rules
    ALB target group health
    Application port
    Health endpoint path
    Security group rules
    Application logs

So, rollout check answered:

    Did Kubernetes deploy the pods successfully?

Smoke test answered:

    Is the application reachable and responding correctly?

---

## Stage 9 — Environment-wise Promotion

After deployment and validation in one environment, the same image tag was promoted to the next environment.

The promotion flow was:

    Dev -> Test -> Stage -> Prod

We did not rebuild the image between environments.

The same Git SHA image moved forward.

### Dev Environment

Dev deployment was mostly automatic after CI success.

Flow:

    CI success
    Image pushed to ECR
    Jenkins connects to Dev AWS account
    Jenkins updates Dev EKS deployment
    Jenkins validates rollout
    Jenkins runs smoke test
    Jenkins notifies team

For dev, deployment was mostly automatic.

After CI pushed the Git SHA tagged image, Jenkins connected to the Dev AWS account and Dev EKS cluster, deployed the image using kubectl, waited for rollout status, and ran smoke tests.

### Test Environment

Test deployment happened after dev validation.

Flow:

    Dev success
    Same image promoted to Test
    Jenkins connects to Test AWS account
    Jenkins updates Test EKS deployment
    QA or automation validates the release

For test, we promoted the same Git SHA image that was already validated in dev.

Jenkins connected to the Test AWS account and Test EKS cluster, deployed the same image with test-specific configuration, and QA or automation tests validated it.

### Stage Environment

Stage was production-like.

Flow:

    Test validation complete
    Same image promoted to Stage
    Jenkins connects to Stage AWS account
    Jenkins updates Stage EKS deployment
    Jenkins validates rollout, ingress, logs, and smoke tests

Stage was treated as a production-like environment.

Jenkins connected to the Stage AWS account and deployed the same image tag with stage-specific configuration.

We validated rollout, ingress routing, logs, and smoke tests before production.

### Production Environment

Production required manual approval.

Flow:

    Stage validation complete
    Manual approval
    Jenkins connects to Prod AWS account
    Jenkins updates Prod EKS deployment
    Jenkins validates rollout
    Jenkins runs smoke test
    Jenkins/team monitors logs, metrics, and alarms
    Rollback if required

For production, Jenkins had a manual approval gate.

After approval, Jenkins connected to the Prod AWS account and Prod EKS cluster, deployed the same Git SHA image using kubectl, waited for rollout status, ran smoke tests, and monitored logs, metrics, pod health, and alarms.

If validation failed, rollback was performed.

The important point is:

    Same image was promoted across environments.
    AWS account and configuration changed per environment.

## Stage 10 — Rollback Handling

Rollback was used when a deployment failed or caused an issue after release.

Rollback means moving the Kubernetes Deployment back to the previous stable version.

Because images were tagged with Git SHA, rollback was traceable and reliable.

Rollback may be needed when:

    Rollout status fails
    Smoke test fails
    Application enters CrashLoopBackOff
    Readiness probe fails
    High 5xx errors after deployment
    Latency increases after deployment
    Critical functionality breaks
    Production alarms trigger

Rollback flow:

    Deployment failed
    Jenkins detects rollout or smoke test failure
    DevOps checks pod status, events, and logs
    Rollback decision is taken
    Deployment is reverted to previous stable image
    Rollout is validated again
    Smoke test is executed again
    Team is notified

Commands used:

    kubectl rollout history deployment/auth-service -n shopease-prod

    kubectl rollout undo deployment/auth-service -n shopease-prod

Rollback to a specific revision:

    kubectl rollout undo deployment/auth-service \
      --to-revision=3 \
      -n shopease-prod

Or redeploy previous stable image:

    kubectl set image deployment/auth-service \
      auth-service=123456789012.dkr.ecr.ap-south-1.amazonaws.com/auth-service:previous-git-sha \
      -n shopease-prod

After rollback, Jenkins or DevOps validated:

    kubectl rollout status deployment/auth-service -n shopease-prod

    kubectl get pods -n shopease-prod -l app=auth-service

    curl -f https://prod.example.com/api/auth/health

For lower environments, rollback can be automated.

For production, rollback was usually controlled.

In production, we usually did not blindly auto-rollback for every failure unless there was an agreed policy.

Jenkins collected rollout history, pod status, events, and logs.

Then DevOps or release owner decided whether to rollback.

For database changes, rollback needed extra care.

Application rollback is easy because we can move back to the previous image.

Database rollback is more sensitive.

For database changes, we preferred backward-compatible migrations and avoided destructive schema changes in the same release.

Important point:

    Container rollback is fast, but database rollback must be planned carefully.

---

## Stage 11 — Notify / Release Summary

At the end of the CD pipeline, Jenkins sent deployment notifications.

Notification was sent for both success and failure.

The notification included:

    Service name
    Target environment
    AWS account
    EKS cluster
    Namespace
    Image tag
    Image URI
    Git SHA
    Deployment status
    Rollout status
    Smoke test result
    Change ticket
    Jenkins build URL
    Rollback status, if rollback happened

This gave visibility to Dev, QA, DevOps, release managers, and stakeholders.

It also helped during audit and troubleshooting.

If deployment failed, notification helped the team quickly identify what failed and where to check logs.

If deployment passed, notification confirmed which image version was deployed to which environment.

## Jenkins Secrets Management in CD

We never hardcoded secrets in Jenkinsfile, Git repository, Dockerfile, or Kubernetes YAML.

Jenkins secrets were managed using Jenkins Credentials Manager.

Examples of Jenkins credentials:

    Git token
    SonarQube token
    Registry credential, if required
    SMTP credential
    API token
    Service account credential

Secrets were injected only when required using withCredentials.

Example:

    withCredentials([string(credentialsId: 'api-token', variable: 'API_TOKEN')]) {
        sh '''
            set +x
            curl -H "Authorization: Bearer $API_TOKEN" https://api.example.com/health
        '''
    }

set +x prevents shell commands from printing secret values in Jenkins logs.

For AWS, we preferred IAM role-based access instead of static access keys.

If Jenkins was running on EC2, we used an IAM instance profile.

If Jenkins agents were running on EKS, we used IRSA.

This avoided long-lived AWS access keys in Jenkins.

We validated AWS identity using:

    aws sts get-caller-identity

Application runtime secrets were not stored directly in Jenkins.

Examples:

    Database password
    JWT secret
    API keys
    OAuth client secret

These were stored in AWS Secrets Manager or AWS Systems Manager Parameter Store.

For Kubernetes, they were injected using External Secrets Operator or Kubernetes Secrets.

Jenkins credentials were used only for CI/CD execution.

Application runtime secrets were managed separately in AWS and injected into Kubernetes at runtime.

---

## Final CD Flow

Select Release
→ Connect to Target AWS Account and EKS Cluster
→ Verify Image in ECR
→ Environment Pre-checks
→ Manual Approval Gate for Stage/Prod
→ Deploy to Kubernetes using kubectl set image
→ Validate Deployment using kubectl rollout status
→ Run Smoke Test
→ Promote Same Image Across Environments
→ Rollback if Required
→ Notify / Release Summary

## Final Short Interview Answer

In our project, the CD pipeline was also implemented in Jenkins.

After CI completed, the Docker image was pushed to ECR with a Git SHA tag.

CD did not rebuild the application.

It picked the already validated image from ECR and deployed it to the target environment.

The CD job was parameterized.

We selected the service name, target environment, image tag, and change ticket if required.

Dev, test, stage, and prod were separate AWS accounts.

Based on the selected environment, Jenkins connected to the respective AWS account using an environment-specific deployment role and updated kubeconfig for that account’s EKS cluster.

Before deployment, Jenkins verified that the selected image tag existed in ECR and performed Kubernetes pre-checks like namespace, deployment, service, ConfigMap, Secret, and RBAC permission validation.

For dev and test, deployments were mostly automated.

For stage and prod, we used Jenkins manual approval gates using the input step.

Production also required a change ticket or release reference.

We were not using Helm for application deployment.

Jenkins deployed using kubectl.

For normal releases, Jenkins updated the image in the existing Kubernetes Deployment using kubectl set image.

The base Kubernetes resources like Deployment, Service, ConfigMap, Secret, Ingress, HPA, PDB, and ServiceAccount were already present in each environment.

After deployment, Jenkins waited for kubectl rollout status, checked pods and service endpoints, and ran smoke tests using health endpoints or critical APIs.

If rollout or smoke test failed, we checked pod logs, events, describe output, and rollout history.

Rollback was done using kubectl rollout undo or by redeploying the previous stable Git SHA image tag.

Finally, Jenkins sent deployment notification with service name, environment, AWS account, image tag, Git SHA, deployment status, smoke test result, rollback status, and Jenkins build URL.

## Very Short Version

CI created a trusted Git SHA tagged image and pushed it to ECR.

CD picked that image, connected to the target AWS account and EKS cluster, verified prerequisites, deployed using kubectl set image, validated rollout, ran smoke tests, used manual approval for stage/prod, and rolled back using kubectl rollout undo or previous stable image tag if deployment failed.
