# Jenkins CD Pipeline Interview Story — AWS DevOps Role

## Overview

In our project, we had a Jenkins-based CD pipeline for deploying Java/Spring Boot microservices to Kubernetes running on Amazon EKS.

The CI pipeline was responsible for building the application, running scans and tests, creating the Docker image, scanning the image, and pushing the trusted image to Amazon ECR with a Git SHA tag.

The CD pipeline started after that.

The main purpose of the CD pipeline was to take the already-built and already-scanned Docker image from ECR and deploy it safely across environments like dev, test, stage, and production.

We followed the build once, deploy many principle.

That means CI created one trusted image, and CD promoted the same image across environments.

We did not rebuild the application separately for each environment.

---

## Important CD Concept

The image remained the same across environments.

Only environment-specific configuration changed.

Example:

```text
auth-service:a1b2c3d -> Dev AWS account
auth-service:a1b2c3d -> Test AWS account
auth-service:a1b2c3d -> Stage AWS account
auth-service:a1b2c3d -> Prod AWS account
```

The configuration changed per environment:

```text
Dev   -> Dev database, Dev secrets, Dev ALB, Dev EKS cluster
Test  -> Test database, Test secrets, Test ALB, Test EKS cluster
Stage -> Stage database, Stage secrets, Stage ALB, Stage EKS cluster
Prod  -> Prod database, Prod secrets, Prod ALB, Prod EKS cluster
```

In our setup, dev, test, stage, and prod were separate AWS accounts.

So Jenkins had to connect to the correct AWS account before deploying to that environment.

---

## Final CD Flow

The CD pipeline had five main stages:

```text
1. Select Release
2. Connect to Target Environment
3. Deploy to Kubernetes
4. Validate Deployment
5. Notify / Rollback
```

This flow was simple, practical, and easy to explain in interviews.

# Stage 1 — Select Release

## What this stage does

The first stage in the CD pipeline was Select Release.

In this stage, Jenkins selected what exactly needed to be deployed.

The CD job accepted parameters like:

```text
Service name
Target environment
Image tag
Change ticket, if required
```

Example:

```text
Service      : auth-service
Environment  : dev / test / stage / prod
Image tag    : a1b2c3d
Image URI    : 123456789012.dkr.ecr.ap-south-1.amazonaws.com/auth-service:a1b2c3d
```

The image tag was usually the Git SHA generated by the CI pipeline.

This helped us trace every deployment back to the exact Git commit and Jenkins CI build.

## Why this stage comes first

This stage comes first because Jenkins must know:

```text
Which service to deploy?
Which image tag to deploy?
Which environment to deploy into?
```

Without these values, the CD pipeline cannot decide the target AWS account, EKS cluster, namespace, or deployment name.

## What happens technically

1. Jenkins CD job is triggered manually or automatically after CI.
2. User selects the target environment.
3. User selects the service name.
4. User provides the Git SHA image tag from CI.
5. Jenkins prepares the image URI.
6. Jenkins maps the selected environment to AWS account, EKS cluster, namespace, and base URL.

## Jenkins parameter example

```groovy
parameters {
    choice(
        name: 'TARGET_ENV',
        choices: ['dev', 'test', 'stage', 'prod'],
        description: 'Select target environment'
    )

    choice(
        name: 'SERVICE_NAME',
        choices: ['auth-service', 'product-service', 'cart-service', 'order-service'],
        description: 'Select service to deploy'
    )

    string(
        name: 'IMAGE_TAG',
        defaultValue: '',
        description: 'Git SHA image tag generated by CI'
    )

    string(
        name: 'CHANGE_TICKET',
        defaultValue: '',
        description: 'Required for stage/prod deployment'
    )
}
```

## What I worked on

In this stage, I worked on making the CD job parameterized so that the same Jenkins pipeline could be used to deploy different services to different environments.

Instead of creating separate deployment jobs for every service and every environment, the pipeline accepted values like service name, image tag, and target environment.

This made the deployment process reusable and traceable.

## Real-time issues

One common issue was the wrong image tag being passed during deployment.

To avoid confusion, we printed important values in Jenkins logs before deployment, such as service name, environment, namespace, image tag, image URI, AWS account, and EKS cluster.

Another issue was missing image tag.

In that case, the pipeline failed early and asked the user to provide the Git SHA image tag generated by CI.

## Interview summary

The Select Release stage decides which service, image tag, and environment will be deployed.

# Stage 2 — Connect to Target Environment

## What this stage does

In our setup, each environment had a separate AWS account.

Example:

```text
Dev   -> Dev AWS account
Test  -> Test AWS account
Stage -> Stage AWS account
Prod  -> Prod AWS account
```

So before deploying, Jenkins had to connect to the correct AWS account and the correct EKS cluster.

This stage selected the environment-specific AWS account, IAM role, EKS cluster, namespace, and application URL.

## What happens technically

1. Jenkins reads the selected target environment.
2. Jenkins maps the environment to the correct AWS account.
3. Jenkins assumes the environment-specific IAM role.
4. Jenkins validates AWS identity using STS.
5. Jenkins updates kubeconfig for the target EKS cluster.
6. Jenkins prepares namespace and base URL for deployment and smoke test.

## AWS identity validation

We validated AWS identity using:

```bash
aws sts get-caller-identity
```

This confirmed that Jenkins was using the correct AWS account and role.

## Updating kubeconfig

Example command:

```bash
aws eks update-kubeconfig \
  --region ap-south-1 \
  --name shopease-dev-eks
```

For separate AWS accounts, Jenkins first assumed the correct role and then updated kubeconfig for that account’s EKS cluster.

## Why this stage is important

This stage is important because deploying to the wrong AWS account or wrong EKS cluster is a serious production risk.

For example, if the user selected dev but Jenkins connected to prod, the wrong environment could be updated.

So we validated AWS identity and printed account and cluster details before deployment.

## What I worked on

I worked on environment mapping and validation logic in the CD pipeline.

For each environment, Jenkins resolved the AWS account, deployment role, EKS cluster, namespace, and base URL.

We validated the AWS identity using aws sts get-caller-identity before running kubectl commands.

## Real-time issues

One common issue was Jenkins using the wrong AWS role or account.

To troubleshoot, I checked the output of:

```bash
aws sts get-caller-identity
```

I verified the AWS account ID, role ARN, and region.

Another issue was kubeconfig pointing to the wrong cluster.

I checked the current context using:

```bash
kubectl config current-context
```

Then I re-ran aws eks update-kubeconfig for the correct cluster.

## Interview summary

Since each environment had a separate AWS account, Jenkins first assumed the correct environment deployment role and connected to that account’s EKS cluster before deploying.

# Stage 3 — Image Verification

## What this stage does

Before deploying to Kubernetes, Jenkins verified that the selected image tag existed in ECR.

This prevented Kubernetes deployment failures caused by wrong image tags, missing images, wrong repository names, or wrong AWS regions.

## Why this stage is important

If Jenkins deploys a non-existing image tag, Kubernetes pods will fail with errors like:

```text
ImagePullBackOff
ErrImagePull
image not found
unauthorized
```

So we checked the image in ECR before touching the Kubernetes deployment.

## Example command

```bash
aws ecr describe-images \
  --region ap-south-1 \
  --repository-name auth-service \
  --image-ids imageTag=a1b2c3d
```

## What I worked on

I supported the image verification step by checking whether the selected Git SHA image existed in ECR before deployment.

This helped us fail early if the image tag was wrong or CI had not pushed the image successfully.

## Real-time issues

One issue was that the image tag provided in the CD job did not exist in ECR.

In that case, I checked:

```text
CI pipeline status
ECR repository name
AWS region
AWS account
Image tag
Jenkins logs from Publish stage
```

Another issue was image present in one AWS account but deployment expected it in another account.

This depends on the ECR pattern used by the organization.

## ECR patterns

There are two common enterprise patterns.

### Pattern 1 — Central ECR

CI pushes the image to a central/shared ECR account.

Environment accounts pull the image from central ECR using cross-account permissions.

```text
Central ECR
   -> Dev EKS pulls image
   -> Test EKS pulls image
   -> Stage EKS pulls image
   -> Prod EKS pulls image
```

### Pattern 2 — Per-environment ECR

Each environment account has its own ECR repository.

The image is promoted or replicated to each environment account’s ECR.

```text
Dev ECR
Test ECR
Stage ECR
Prod ECR
```

In interview, we can say:

The image was tagged with Git SHA and either pulled from a central ECR using cross-account access or promoted to the target account’s ECR based on environment policy.

## Interview summary

Image Verification confirms that the selected Git SHA image exists in ECR before Kubernetes deployment starts.

# Stage 4 — Environment Pre-checks

## What this stage does

Before deploying the image, Jenkins performed basic Kubernetes pre-checks.

The purpose was to make sure the target environment was ready for deployment.

## Common checks

Jenkins checked:

```text
Namespace exists
Deployment exists
Service exists
ConfigMap exists
Secret exists
Jenkins has RBAC permission
EKS cluster is reachable
Kubernetes context is correct
```

## Example commands

```bash
kubectl get namespace shopease-dev

kubectl get deployment auth-service -n shopease-dev

kubectl get service auth-service -n shopease-dev

kubectl get configmap -n shopease-dev

kubectl get secret -n shopease-dev

kubectl auth can-i patch deployment -n shopease-dev
```

## Why this stage is important

This stage helps avoid deployment failure due to missing Kubernetes resources or permission issues.

For example, if Jenkins does not have permission to patch deployment, the pipeline should fail before trying to deploy.

## What I worked on

I supported the pre-check stage by validating namespace, deployment, service, ConfigMap, Secret, and Jenkins RBAC permissions before deployment.

This helped us catch environment-level issues before updating the running workload.

## Real-time issues

One common issue was RBAC permission failure.

The deployment failed because Jenkins did not have permission to patch deployments in the target namespace.

I checked:

```bash
kubectl auth can-i patch deployment -n shopease-dev
```

Then I verified Role, RoleBinding, ClusterRole, or service account mapping.

Another issue was missing ConfigMap or Secret.

In that case, the new pod could fail during startup.

I checked the Deployment YAML and verified required environment variables and mounted secrets.

## Interview summary

Environment Pre-checks validate Kubernetes access, required resources, and RBAC permissions before deployment.

# Stage 5 — Manual Approval Gate

## What this stage does

Manual approval gates were used for higher environments like stage and production.

Dev and test deployments were mostly automated.

Stage could have optional approval depending on release process.

Production had mandatory manual approval.

## Environment behavior

```text
Dev   -> automatic deployment
Test  -> automatic or QA-controlled deployment
Stage -> approval or release validation
Prod  -> mandatory manual approval
```

## How manual gate was implemented

Manual gates were implemented using a parameterized Jenkins CD pipeline and Jenkins input step.

Parameterized pipeline selected:

```text
Service name
Target environment
Image tag
Change ticket
```

Jenkins input step paused the pipeline before deploying to stage or production.

Only after approval, Jenkins continued to the deployment stage.

## Jenkins approval example

```groovy
stage('Manual Approval') {
    when {
        expression {
            params.TARGET_ENV == 'stage' || params.TARGET_ENV == 'prod'
        }
    }
    steps {
        input message: "Approve deployment of ${params.SERVICE_NAME}:${params.IMAGE_TAG} to ${params.TARGET_ENV}?",
              ok: "Approve Deployment"
    }
}
```

## Production approval with restricted approvers

```groovy
stage('Production Approval') {
    when {
        expression { params.TARGET_ENV == 'prod' }
    }
    steps {
        input message: "Approve production deployment for ${params.SERVICE_NAME}:${params.IMAGE_TAG}?",
              ok: "Deploy to Production",
              submitter: "release-managers,devops-leads"
    }
}
```

## Change ticket validation

For stage and prod, we also required a change ticket or release reference.

Example:

```groovy
stage('Validate Change Ticket') {
    when {
        expression {
            params.TARGET_ENV == 'stage' || params.TARGET_ENV == 'prod'
        }
    }
    steps {
        script {
            if (!params.CHANGE_TICKET?.trim()) {
                error "CHANGE_TICKET is required for stage/prod deployment"
            }

            echo "Change Ticket: ${params.CHANGE_TICKET}"
        }
    }
}
```

## What I worked on

I worked on making the CD pipeline parameterized and adding approval gates for higher environments.

This ensured that production deployments did not happen automatically without review.

## Real-time issues

One issue was users forgetting to provide the change ticket for stage or prod deployment.

To handle that, the pipeline validated CHANGE_TICKET before approval.

Another issue was unauthorized users approving production deployment.

To avoid this, Jenkins input approval could be restricted to specific users or groups.

## Interview summary

Manual gates were implemented using Jenkins parameterized pipeline and input approval step.

Dev/test deployments were automated, while stage/prod required approval and change-ticket validation.

# Stage 6 — Deploy to Kubernetes

## What this stage does

This is the actual deployment stage.

We were not using Helm for application deployment.

Jenkins deployed using kubectl.

For normal releases, Jenkins updated the image in the existing Kubernetes Deployment using kubectl set image.

## Important point

Kubernetes resources were already present in the cluster.

Existing resources included:

```text
Deployment
Service
ConfigMap
Secret
Ingress
HPA
PDB
ServiceAccount
```

For normal releases, Jenkins only updated the image tag.

## Example command

```bash
kubectl set image deployment/auth-service \
  auth-service=123456789012.dkr.ecr.ap-south-1.amazonaws.com/auth-service:a1b2c3d \
  -n shopease-dev
```

## What happens during deployment

Before deployment:

```text
auth-service:old123
```

After deployment:

```text
auth-service:a1b2c3d
```

Kubernetes then creates a new ReplicaSet and starts new pods using the new image.

Old pods are gradually replaced based on RollingUpdate strategy.

## RollingUpdate strategy

Example:

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0
```

## Explanation

maxUnavailable 0 means Kubernetes should not intentionally reduce available pods during rollout.

maxSurge 1 means Kubernetes can create one extra pod temporarily during rollout.

Readiness probe ensures that traffic goes only to healthy new pods.

## What if manifest changes are needed?

For normal application releases, kubectl set image was enough.

But if there were manifest-level changes like resource limits, probes, ConfigMap, HPA, PDB, or Ingress changes, then we used kubectl apply after review.

Example:

```bash
kubectl apply -f deployment-kubernetes/auth-service/ -n shopease-dev
```

## What I worked on

I supported kubectl-based deployments from Jenkins.

My responsibility was to ensure the correct image tag was deployed to the correct environment, namespace, and deployment.

I also checked deployment issues related to wrong container name, wrong namespace, wrong image URI, and missing Kubernetes resources.

## Real-time issues

One common issue was container name mismatch.

kubectl set image requires the correct container name from the Deployment.

To troubleshoot, I checked:

```bash
kubectl describe deployment auth-service -n shopease-dev
```

or checked the Deployment YAML.

Another issue was Jenkins deploying to the wrong namespace.

To avoid that, we printed namespace and environment details before deployment.

## Interview summary

Deploy stage updated the Kubernetes Deployment with the selected Git SHA image using kubectl set image.

# Stage 7 — Validate Deployment

## What this stage does

After deployment, Jenkins validated whether the rollout completed successfully.

The main command was:

```bash
kubectl rollout status deployment/auth-service -n shopease-dev --timeout=5m
```

This command waits until the new ReplicaSet is successfully rolled out.

## What rollout validation confirms

Rollout validation confirms:

```text
New ReplicaSet is created
New pods are scheduled
New pods become ready
Old pods are replaced safely
Deployment reaches desired state
```

## Additional checks

After rollout status, Jenkins checked:

```bash
kubectl get pods -n shopease-dev -l app=auth-service

kubectl get service auth-service -n shopease-dev

kubectl get endpoints auth-service -n shopease-dev
```

## Common deployment failures

Rollout validation can catch:

```text
ImagePullBackOff
ErrImagePull
CrashLoopBackOff
Readiness probe failure
Liveness probe failure
Missing ConfigMap
Missing Secret
Wrong environment variable
Insufficient CPU/memory
Application startup failure
```

## What I worked on

I worked on validating rollout status after deployment.

If rollout failed, I checked pod status, events, describe output, and logs.

## Troubleshooting commands

```bash
kubectl get pods -n shopease-dev

kubectl describe pod <pod-name> -n shopease-dev

kubectl logs <pod-name> -n shopease-dev

kubectl describe deployment auth-service -n shopease-dev

kubectl get events -n shopease-dev --sort-by=.lastTimestamp
```

## Real-time issues

One common issue was ImagePullBackOff.

In that case, I checked whether the image existed in ECR, whether the image URI was correct, whether the EKS nodes had permission to pull from ECR, and whether the correct AWS account/region was used.

Another common issue was CrashLoopBackOff.

I checked application logs, environment variables, database connectivity, secrets, and application startup errors.

Another issue was readiness probe failure.

I checked the health endpoint path, application startup time, port configuration, and readiness probe settings.

## Interview summary

Validate Deployment stage confirms that the Kubernetes rollout completed successfully and new pods became ready.

# Stage 8 — Smoke Test

## What this stage does

After Kubernetes rollout completed, Jenkins ran smoke tests.

Smoke tests are lightweight tests that verify whether the application is responding correctly after deployment.

## Difference between rollout check and smoke test

Rollout check answers:

```text
Did Kubernetes deploy the pods successfully?
```

Smoke test answers:

```text
Is the application actually responding from user or service point of view?
```

## Example smoke test

```bash
curl -f https://dev.example.com/api/auth/health
```

Service-specific examples:

```text
auth-service    -> /api/auth/health
product-service -> /api/products/health
cart-service    -> /api/cart/health
order-service   -> /api/orders/health
```

## What I worked on

I supported smoke test validation after deployment.

We checked health endpoints and basic API availability to confirm that the service was reachable after deployment.

## Real-time issues

One common issue was rollout succeeded, but smoke test failed.

This means Kubernetes pods were running, but the application was not reachable through service, ingress, or ALB.

In that case, I checked:

```text
Service selector
Service endpoints
Ingress rules
ALB target group health
Application port
Health endpoint path
Security group rules
Application logs
```

## Interview summary

Smoke Test validates that the deployed application is reachable and responding correctly after Kubernetes rollout.

# Stage 9 — Environment-wise Promotion

## What this stage does

After deployment and validation in one environment, the same image tag was promoted to the next environment.

The promotion flow was:

```text
Dev -> Test -> Stage -> Prod
```

We did not rebuild the image between environments.

The same image tag moved forward.

## Dev environment

Dev deployment was mostly automatic after CI success.

Flow:

```text
CI success
Image pushed to ECR
Jenkins connects to Dev AWS account
Jenkins updates Dev EKS deployment
Jenkins validates rollout
Jenkins runs smoke test
Jenkins notifies team
```

Interview explanation:

For dev, deployment was mostly automatic.

After CI pushed the Git SHA tagged image, Jenkins connected to the Dev AWS account and Dev EKS cluster, deployed the image using kubectl, waited for rollout status, and ran smoke tests.

## Test environment

Test deployment happened after dev validation.

Flow:

```text
Dev success
Same image promoted to Test
Jenkins connects to Test AWS account
Jenkins updates Test EKS deployment
QA or automation validates the release
```

Interview explanation:

For test, we promoted the same Git SHA image that was already validated in dev.

Jenkins connected to the Test AWS account and Test EKS cluster, deployed the same image with test-specific configuration, and QA or automation tests validated it.

## Stage environment

Stage was production-like.

Flow:

```text
Test validation complete
Same image promoted to Stage
Jenkins connects to Stage AWS account
Jenkins updates Stage EKS deployment
Jenkins validates rollout, ingress, logs, and smoke tests
```

Interview explanation:

Stage was treated as a production-like environment.

Jenkins connected to the Stage AWS account and deployed the same image tag with stage-specific configuration.

We validated rollout, ingress routing, logs, and smoke tests before production.

## Prod environment

Production required manual approval.

Flow:

```text
Stage validation complete
Manual approval
Jenkins connects to Prod AWS account
Jenkins updates Prod EKS deployment
Jenkins validates rollout
Jenkins runs smoke test
Jenkins/team monitors logs, metrics, and alarms
Rollback if required
```

Interview explanation:

For production, Jenkins had a manual approval gate.

After approval, Jenkins connected to the Prod AWS account and Prod EKS cluster, deployed the same Git SHA image using kubectl, waited for rollout status, ran smoke tests, and monitored logs, metrics, pod health, and alarms.

If validation failed, rollback was performed.

## Interview summary

The same image was promoted across dev, test, stage, and prod.

Since each environment had a separate AWS account, Jenkins connected to the respective account and EKS cluster before deployment.

# Stage 10 — Rollback Handling

## What this stage does

Rollback is used when a deployment fails or causes issues after release.

Rollback means moving the Kubernetes Deployment back to the previous stable version.

Since our images were tagged with Git SHA, rollback was reliable and traceable.

## When rollback is needed

Rollback may be needed when:

```text
Rollout status fails
Smoke test fails
Application enters CrashLoopBackOff
Readiness probe fails
High 5xx errors after deployment
Latency increases after deployment
Critical functionality breaks
Production alarms trigger
```

## Rollback flow

```text
Deployment failed
Jenkins detects rollout or smoke test failure
DevOps checks pod status, events, and logs
Rollback decision is taken
Deployment is reverted to previous stable image
Rollout is validated again
Smoke test is executed again
Team is notified
```

## Rollback commands

Check rollout history:

```bash
kubectl rollout history deployment/auth-service -n shopease-prod
```

Rollback to previous ReplicaSet:

```bash
kubectl rollout undo deployment/auth-service -n shopease-prod
```

Rollback to a specific revision:

```bash
kubectl rollout undo deployment/auth-service \
  --to-revision=3 \
  -n shopease-prod
```

Redeploy previous stable image:

```bash
kubectl set image deployment/auth-service \
  auth-service=123456789012.dkr.ecr.ap-south-1.amazonaws.com/auth-service:previous-git-sha \
  -n shopease-prod
```

Validate rollback:

```bash
kubectl rollout status deployment/auth-service -n shopease-prod

kubectl get pods -n shopease-prod -l app=auth-service

curl -f https://prod.example.com/api/auth/health
```

## Manual rollback vs automatic rollback

For lower environments, rollback can be automated.

For production, rollback is usually controlled.

In production, we usually do not blindly auto-rollback for every failure unless there is an agreed policy.

Jenkins can collect rollout history, pod status, events, and logs.

Then DevOps or release owner can decide whether to rollback.

## What I worked on

I supported rollback by checking rollout status, pod logs, Kubernetes events, and deployment history.

If the issue was due to the new application version, we rolled back using kubectl rollout undo or redeployed the previous stable Git SHA image.

## Real-time issues

One issue was CrashLoopBackOff after deployment.

I checked pod logs and found that the application failed during startup due to wrong configuration or missing secret.

If the issue impacted the environment, we rolled back to the previous stable version.

Another issue was smoke test failure after rollout success.

In that case, I checked ingress, service endpoints, ALB target health, and application logs.

If the new version was causing the issue, rollback was performed.

## Database rollback note

Application rollback is easier because we can move back to the previous image.

Database rollback is more sensitive.

For database changes, we preferred backward-compatible migrations.

We avoided destructive schema changes in the same release.

If database rollback was required, it needed a planned rollback script or restore strategy and proper approval.

Important point:

```text
Container rollback is fast, but database rollback must be planned carefully.
```

## Interview summary

Rollback was done by reverting the Kubernetes Deployment to the previous stable image using kubectl rollout undo or kubectl set image with the previous Git SHA tag.

# Jenkins Secrets Management in CD

## How secrets were managed in Jenkins

We never hardcoded secrets in Jenkinsfile, Git repository, Dockerfile, or Kubernetes YAML.

Jenkins secrets were managed using Jenkins Credentials Manager.

Examples of Jenkins credentials:

```text
Git token
SonarQube token
ECR or registry credential, if required
SMTP credential
API token
Service account credential
```

## Safe usage in Jenkinsfile

Secrets were injected only when required using withCredentials.

Example:

```groovy
withCredentials([string(credentialsId: 'api-token', variable: 'API_TOKEN')]) {
    sh '''
        set +x
        curl -H "Authorization: Bearer $API_TOKEN" https://api.example.com/health
    '''
}
```

## Why set +x is used

set +x prevents shell commands from printing secret values in Jenkins logs.

## AWS access secrets

For AWS, we preferred IAM role-based access instead of static access keys.

If Jenkins was running on EC2, we used an IAM instance profile.

If Jenkins agents were running on EKS, we used IRSA.

This avoided long-lived AWS access keys in Jenkins.

We validated AWS identity using:

```bash
aws sts get-caller-identity
```

## Application runtime secrets

Application runtime secrets were not stored directly in Jenkins.

Examples:

```text
Database password
JWT secret
API keys
OAuth client secret
```

These were stored in:

```text
AWS Secrets Manager
AWS Systems Manager Parameter Store
```

For Kubernetes, they were injected using:

```text
External Secrets Operator
Kubernetes Secrets
```

## Interview summary

Jenkins credentials were used only for CI/CD execution.

Application runtime secrets were managed separately in AWS Secrets Manager or Parameter Store and injected into Kubernetes at runtime.

After CI completed, the Docker image was pushed to ECR with a Git SHA tag.

CD did not rebuild the application.

It picked the already validated image from ECR and deployed it to the target environment.

The CD job was parameterized.

We selected the service name, target environment, image tag, and change ticket if required.

Dev, test, stage, and prod were separate AWS accounts.

Based on the selected environment, Jenkins connected to the respective AWS account using an environment-specific deployment role and updated kubeconfig for that account’s EKS cluster.

Before deployment, Jenkins verified that the selected image tag existed in ECR and performed Kubernetes pre-checks like namespace, deployment, service, ConfigMap, Secret, and RBAC permission validation.

For dev and test, deployments were mostly automated.

For stage and prod, we used Jenkins manual approval gates using the input step.

Production also required a change ticket or release reference.

We were not using Helm for application deployment.

Jenkins deployed using kubectl.

For normal releases, Jenkins updated the image in the existing Kubernetes Deployment using kubectl set image.

The base Kubernetes resources like Deployment, Service, ConfigMap, Secret, Ingress, HPA, PDB, and ServiceAccount were already present in each environment.

After deployment, Jenkins waited for kubectl rollout status, checked pods and service endpoints, and ran smoke tests using health endpoints or critical APIs.

If rollout or smoke test failed, we checked pod logs, events, describe output, and rollout history.

Rollback was done using kubectl rollout undo or by redeploying the previous stable Git SHA image tag.

Finally, Jenkins sent deployment notification with service name, environment, AWS account, image tag, Git SHA, deployment status, smoke test result, rollback status, and Jenkins build URL.

# Very Short Version

CI created a trusted Git SHA tagged image and pushed it to ECR.

CD picked that image, connected to the target AWS account and EKS cluster, verified prerequisites, deployed using kubectl set image, validated rollout, ran smoke tests, used manual approval for stage/prod, and rolled back using kubectl rollout undo or previous stable image tag if deployment failed.
